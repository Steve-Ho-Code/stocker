# 專案綜合架構與深度審查報告

## 高階摘要 (Executive Summary)
Stocker 專案是一個目標明確的 Telegram 機器人，其架構在目前規模下尚稱合理。程式碼結構清晰，利用 `python-telegram-bot`、`APScheduler` 和 `Pydantic` 等成熟函式庫，實現了核心功能。專案具備基本的 CI 流程和容器化能力，顯示了良好的 DevOps 早期實踐。

然而，專案在多個關鍵領域存在顯著風險，若不加以解決，將嚴重影響其未來的可擴展性、安全性與可維護性。最主要的隱憂集中在授權機制過於簡陋、缺乏結構化錯誤處理、日誌資訊不足，以及 CI/CD 流程不完整。此外，目前的設定管理方式雖然可行，但在多實例部署或更複雜的組態情境下將面臨挑戰。儘管專案目前功能單一，但這些基礎問題將成為未來發展的技術債。

總體而言，Stocker 的健康度處於「尚可」水平，具備部署能力，但安全性和可維護性有待加強。本報告將提出一系列可執行的建議，旨在強化其架構基礎，為未來的成長做好準備。

## 核心優勢 (Strengths)
* **清晰的專案結構:** 專案遵循了常見的 Python 應用程式佈局，將原始碼 (`src`)、文件 (`docs`) 和測試 (`tests`) 明確分離，易於理解和導航。
* **容器化部署:** 提供 `Dockerfile`，簡化了開發環境的設定和生產環境的部署流程，提升了可移植性。
* **非同步程式設計:** 專案從一開始就採用 `asyncio`，這對於 I/O 密集的機器人應用程式至關重要，為處理高並行請求奠定了良好基礎。
* **設定管理:** 使用 `Pydantic` 進行設定管理，提供了類型檢查和驗證，並將動態設定與環境變數分離，增加了靈活性。

## 隱憂與高風險區域 (Areas of Concern & Risks)
* **[授權機制] (安全):** 目前的授權方式是將授權的使用者 ID 硬編碼在 `.env` 檔案的 `AUTHORIZED_USER_IDS` 字串中，並在 `src/bot/handlers.py` 中手動解析和比對。這種方法極不靈活、難以擴展，且每次變更都需要重啟應用程式。當使用者增加時，管理會變得很麻煩，且容易出錯。
* **[日誌與監控] (可觀測性):** 專案雖然設定了日誌，但僅使用 `StreamHandler` 輸出到控制台，且日誌訊息缺乏結構和上下文（例如，使用者 ID、對話 ID）。在 `src/main.py` 中設定的 `jsonlogger` 並未在所有模組中一致使用，導致難以進行有效的日誌聚合、查詢和警報。
* **[CI/CD 流程] (DevOps):** `.github/workflows/ci.yml` 中的 CI 流程僅包含基本的 linting 和測試。它缺少關鍵的步驟，如：依賴項漏洞掃描 (vulnerability scanning)、Docker 映像檔的建置與推送，以及部署到任何環境的自動化流程。這使得部署過程仍然是手動的，且無法在早期發現潛在的安全風險。
* **[錯誤處理] (可靠性):** 專案缺乏統一的錯誤處理策略。在 `src/bot/handlers.py` 的 `send_price_update` 函式中，API 請求失敗時僅記錄一條錯誤日誌並向使用者傳送一條通用訊息。系統沒有重試機制，也沒有定義明確的錯誤邊界，這可能導致在外部服務不穩定時，機器人服務完全中斷。
* **[設定持久化] (擴展性):** 將動態設定（如 `SYMBOL`）儲存在 `settings.json` 檔案中，雖然在單一實例下可行，但在水平擴展（部署多個機器人實例）時會導致狀態不一致的問題。此外，對該檔案的寫入操作使用了 `asyncio.Lock`，這在單一程序中可以防止競爭條件，但無法處理多程序或多節點的情況。

## 詳細維度分析 (Detailed Analysis)
### 結構組織與領域邊界
專案結構大致清晰，`src` 目錄下的 `bot` 和 `providers` 資料夾劃分了不同的職責。然而，`config.py` 承擔了過多的職責，它不僅載入設定，還包含了更新設定並將其持久化的業務邏輯，這違反了單一職責原則。

### 設計模式與架構一致性
專案使用了 `ConversationHandler` 來管理多步驟的對話流程，這是一個很好的實踐。然而，在 `src/bot/handlers.py` 中，授權邏輯在多個處理函式中重複出現，應將其重構為一個可重複使用的裝飾器 (decorator) 或中介軟體。

### 依賴管理與模組耦合度
`requirements.txt` 中包含了應用程式和測試所需的依賴。建議將其分為 `requirements.in`（用於主要依賴）和 `requirements-dev.txt`（用於開發和測試），並使用 `pip-tools` 來編譯和鎖定版本，以實現更可預測的建置。

### 可擴展性、效能與潛在瓶頸
目前最大的擴展性瓶頸是 `settings.json` 的使用。若要水平擴展，應將動態設定遷移到如 Redis 或資料庫等共享的外部儲存中。`aiocache` 的使用是一個亮點，有效降低了對外部 API 的請求頻率，但其記憶體內的特性在多實例部署下同樣會有效能和一致性問題。

### 安全性與資料保護
授權機制是最大的安全弱點。除了不易管理外，將授權 ID 儲存在環境變數中也非最安全的做法。應考慮使用資料庫來儲存使用者角色和權限。此外，CI 流程中缺乏對依賴項的漏洞掃描，可能引入已知漏洞的風險。

### 可觀測性與錯誤處理
日誌記錄過於基本，缺乏結構化和上下文資訊。錯誤處理不夠健全，例如在 `src/providers/alpha_vantage.py` 中，對 API 回應的解析很脆弱，如果 API 回應的結構發生變化，可能會引發未處理的 `KeyError`。

### DevOps 與 CI/CD
`Dockerfile` 應採用多階段建構 (multi-stage build) 來減小最終映像檔的大小，將建置時的依賴與執行階段的環境分離。CI 流程應擴展，包含安全性掃描、建置 Docker 映像檔並推送到容器儲存庫 (container registry) 的步驟。

### 資料庫與狀態管理
專案目前沒有使用資料庫，這簡化了部署。然而，這也限制了其功能，例如無法實現更複雜的使用者管理、權限系統或持久化使用者特定的設定。將動態設定儲存在 `settings.json` 是一個脆弱的狀態管理形式。

## 可執行的建議 (Actionable Recommendations)
### P0：緊急修復與短期速贏 (Critical Fixes & Short-Term Wins)
* **重構授權邏輯:** 立即將 `src/bot/handlers.py` 中的重複授權檢查重構為一個 Python 裝飾器，以清理程式碼並集中管理權限檢查。
* **增強日誌記錄:** 在所有處理函式和服務中，為日誌訊息添加關鍵上下文，例如使用者 ID 和指令名稱。確保 `jsonlogger` 在整個應用程式中一致使用。
* **修正輸入驗證:** 在 `set_timer` 和 `receive_timer` 函式中，對使用者輸入的 `new_interval` 進行更嚴格的驗證，例如限制其最大值，防止潛在的資源濫用。

### P1：架構與流程優化 (Architecture & Process Improvements)
* **引入依賴項漏洞掃描:** 在 `.github/workflows/ci.yml` 中加入一個步驟，使用 `pip-audit` 或類似工具來掃描 `requirements.txt` 中的已知漏洞。
* **優化 Dockerfile:** 將 `Dockerfile` 修改為多階段建構，以減小生產映像檔的大小。
* **遷移設定儲存:** 將 `settings.json` 中的動態設定遷移到 Redis。這將解決水平擴展時的狀態一致性問題，並為未來的快取和工作佇列需求提供基礎。

### P2：長期策略性變更 (Long-Term Strategic Changes)
* **引入資料庫:** 為了支援更複雜的功能（如使用者角色、權限管理），規劃引入一個輕量級的資料庫（如 SQLite 或 PostgreSQL），並使用 ORM（如 SQLAlchemy）來管理資料模型。
* **建立完整的 CI/CD 流程:** 擴展 GitHub Actions 工作流程，實現從程式碼提交到自動化測試、映像檔建置、推送到容器儲存庫，最終自動部署到預備 (staging) 或生產環境的完整流程。
* **模組解耦:** 將 `config.py` 中的設定更新邏輯分離到一個新的 `services` 或 `repositories` 模組中，使其職責更單一，提高程式碼的可測試性和可維護性。
